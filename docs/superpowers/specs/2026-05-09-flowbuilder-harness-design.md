# Flowbuilder Harness Design

**Status**: Draft
**Date**: 2026-05-09
**Scope**: Harness-side support for an n8n-style flow builder. On-disk session schema, custom tools, plugin hooks. UI integration deferred.

## Goal

Let an agent build and edit a flow graph that the Electron app renders. The agent reads and writes a small JSON file per chat session. The Electron app owns session creation; the harness owns persistence and validation.

## Non-Goals

- UI rendering of the flow graph (separate spec).
- IPC channel implementation between Electron renderer and main (separate spec).
- Runtime execution semantics — how the agent actually runs the graph end-to-end at the user's request.
- Output passing / `$ref` resolution between nodes — agent maps values at runtime.
- Multi-process locking, multi-agent concurrency on one session.
- State migrations across `schemaVersion` bumps.

## Architecture

A new package `@flow-build/flowbuilder` plugs into the existing harness plugin system (`packages/core/src/plugin/*`). It composes with the existing `@flow-build/rote` plugin: rote drives flow discovery and creation via its skill; flowbuilder owns session state on disk.

```
packages/flowbuilder/
  src/
    index.ts          # createFlowbuilderPlugin({ baseDir, sessionId }): Plugin
    session.ts        # SessionManager: paths, atomic IO
    schema.ts         # Zod schemas: Manifest, State, Node, Edge
    tools.ts          # set_state, get_state tool definitions
    prompt.ts         # promptPrefix renderer
    rules.ts          # writes .cursor/rules/.flow-build-flowbuilder.mdc
    errors.ts         # typed FlowbuilderError hierarchy
  test/
    schema.test.ts
    session.test.ts
    tools.test.ts
    plugin.integration.test.ts
  package.json
  tsconfig.json
```

### Lifecycle

```
Electron (renderer)
  user clicks New Chat
        │ IPC: createSession({ name })
        ▼
Electron (main)
  - mkdir <baseDir>/sessions/<id>/
  - write manifest.json
  - write empty state.json
        │ spawn flow-build run \
        │   --session <id> \
        │   --flowbuilder-base <userData>/flowbuilder \
        │   "<user prompt>"
        ▼
flow-build CLI
  registers plugins: rote, flowbuilder({ baseDir, sessionId })
        │
        ▼
Cursor SDK agent
  - reads rote skill, flowbuilder rules
  - reads current state via flowbuilder_get_state
  - calls rote CLI to find/create flows
  - emits new full state via flowbuilder_set_state
        │
        ▼
flowbuilder plugin
  - validate (Zod + referential integrity)
  - tmp write + rename
  - bump manifest.updatedAt
```

### Disk Layout

Base directory: Electron `app.getPath("userData") + "/flowbuilder"`. CLI standalone default: `~/.flowbuilder`. Always passed explicitly via `--flowbuilder-base`.

```
<baseDir>/
  sessions/
    <sessionId>/
      manifest.json
      state.json
```

`<sessionId>` format: `s_` + 12-char crockford-base32. Generated by Electron via `nanoid`.

## Schemas

### `manifest.json`

```json
{
  "schemaVersion": 1,
  "id": "s_8x3k2pq7nw9r",
  "name": "GitHub digest pipeline",
  "description": "",
  "createdAt": "2026-05-09T10:00:00.000Z",
  "updatedAt": "2026-05-09T11:42:13.421Z"
}
```

Zod:

```ts
const ManifestSchema = z.object({
  schemaVersion: z.literal(1),
  id: z.string().regex(/^s_[0-9a-z]{12}$/),
  name: z.string().min(1),
  description: z.string(),
  createdAt: z.string().datetime(),
  updatedAt: z.string().datetime(),
});
```

Tool input/output JSON Schemas are derived from Zod via `zod-to-json-schema` at plugin construction.

### `state.json`

```json
{
  "schemaVersion": 1,
  "nodes": [
    { "id": "n1", "type": "input",  "value": { "owner": "anthropics", "repo": "claude-code" } },
    { "id": "n2", "type": "flow",   "flow": "github/fetch-issues", "params": { "owner": "anthropics", "repo": "claude-code" } },
    { "id": "n3", "type": "branch", "cond": "issues.length > 0" },
    { "id": "n4", "type": "flow",   "flow": "text/summarize", "params": {} },
    { "id": "n5", "type": "output", "value": null }
  ],
  "edges": [
    { "from": "n1", "to": "n2" },
    { "from": "n2", "to": "n3" },
    { "from": "n3", "to": "n4" },
    { "from": "n4", "to": "n5" }
  ]
}
```

Node discriminated union (`type` is the discriminant):

| `type`   | Fields                                                       |
| -------- | ------------------------------------------------------------ |
| `input`  | `id`, `type`, `value: unknown`                               |
| `output` | `id`, `type`, `value: unknown`                               |
| `flow`   | `id`, `type`, `flow: string` (rote `<category>/<name>`), `params: Record<string, unknown>` |
| `branch` | `id`, `type`, `cond: string`                                 |
| `merge`  | `id`, `type`                                                 |

Edge: `{ from: string, to: string }`. No labels.

Zod sketch:

```ts
const NodeSchema = z.discriminatedUnion("type", [
  z.object({ id: z.string(), type: z.literal("input"),  value: z.unknown() }),
  z.object({ id: z.string(), type: z.literal("output"), value: z.unknown() }),
  z.object({ id: z.string(), type: z.literal("flow"),   flow: z.string().regex(/^[^/]+\/[^/]+$/), params: z.record(z.unknown()) }),
  z.object({ id: z.string(), type: z.literal("branch"), cond: z.string() }),
  z.object({ id: z.string(), type: z.literal("merge") }),
]);

const EdgeSchema = z.object({ from: z.string(), to: z.string() });

const StateSchema = z.object({
  schemaVersion: z.literal(1),
  nodes: z.array(NodeSchema),
  edges: z.array(EdgeSchema),
});
```

Params are free-form `Record<string, unknown>`. The agent maps upstream-output to downstream-param at runtime; the file does not encode wiring beyond the edge graph.

Referential integrity (validated on write, not encoded in Zod alone):

- Every `node.id` is unique.
- Every `edge.from` and `edge.to` matches some `node.id`.

## Custom Tools

Both tools close over `sessionId` at plugin instantiation. Session id is **not** a tool parameter.

### `flowbuilder_get_state`

- **Input**: `{}`
- **Output**: `{ ok: true, state: State }` or `{ ok: false, error: string }`
- **Errors**: `invalid_schema: <zod-path>: <message>`, `io: <message>`

### `flowbuilder_set_state`

- **Input**: `{ state: State }` — full state, no patches.
- **Action**:
  1. Validate `state` via Zod.
  2. Validate referential integrity (unique ids, edge endpoints exist).
  3. Atomic write `state.json.<runId>.tmp` in the session dir, then `fs.rename` to `state.json`.
  4. Atomic-bump `manifest.updatedAt` via the same tmp+rename pattern.
- **Output**: `{ ok: true, bytes: number, updatedAt: string }`
- **Errors**: `validation: <zod-path>: <message>`, `ref_integrity: <message>`, `io: <message>`

Tool definitions register with the Cursor SDK with JSON Schema generated from Zod. Tool names follow the existing rote convention (`<adapter>_<verb>`).

## Plugin Hooks

`createFlowbuilderPlugin({ baseDir, sessionId })`. Both arguments are required; the TypeScript signature rejects undefined. There is no fallback path for a missing session.

| Hook             | Behavior                                                     |
| ---------------- | ------------------------------------------------------------ |
| `preRun`         | Resolve `<baseDir>/sessions/<sessionId>/`. Stat manifest + state. Missing → `FlowbuilderSessionMissingError`. Parse + validate both. Stash `{ sessionId, sessionDir, manifest, state }` in `ctx.state.set("flowbuilder", ...)`. |
| `systemPrompt`   | Write `.cursor/rules/.flow-build-flowbuilder.mdc`: schema reference, two-tool contract, "session is fixed for this run", "always emit FULL state via set_state — never partial". |
| `promptPrefix`   | Render the active-session digest (see below).                |
| `interceptEvent` | Pass-through.                                                |
| `onToolCall`     | Log writes via `ctx.logger`.                                 |
| `cleanup`        | Nothing — session files persist across runs by design.       |

`promptPrefix` output:

```
[flowbuilder] active session: s_8x3k2pq7nw9r
manifest: name="GitHub digest pipeline" updated=2026-05-09T11:42Z
current state: 4 nodes, 3 edges
call flowbuilder_get_state to read full state.
```

There is no missing-session branch. If the session is missing, `preRun` throws and the agent never starts.

## Composition With Rote Plugin

Both plugins coexist. Plugin registration order in `packages/cli/src/main.ts`:

1. `createRotePlugin(...)` — provides flow discovery / creation context, exposes the rote skill to the agent.
2. `createFlowbuilderPlugin({ baseDir, sessionId })` — provides session context and the two custom tools.

No shared `ctx.state` keys (`"rote"`, `"rote:lastToolArgs"` vs `"flowbuilder"`). The agent uses the rote skill to find or create flows and the flowbuilder tools to record nodes that reference them.

## CLI Wiring

Extend `flow-build run`:

- `--session <id>` — required when flowbuilder is active (always).
- `--flowbuilder-base <dir>` — required when flowbuilder is active.

Both flags are required at all times; missing either is a CLI usage error before the harness starts. There is **no** `FLOW_BUILD_DISABLE_FLOWBUILDER` env var. Flowbuilder runs in every invocation.

Electron passes both flags via `child_process.spawn` args.

## Atomic Writes

- Write to `<file>.<runId>.tmp` in the same directory.
- `fs.rename` to the destination. POSIX `rename(2)` is atomic on a single filesystem.
- Same pattern for `state.json` and `manifest.json`.
- Windows: `fs.promises.rename` is atomic enough for our use. Power-loss durability is not guaranteed and not in scope.

## Concurrency

- Contract: one agent process per session at a time; Electron enforces.
- Renderer is read-only on these files. UI watcher debounces ~200ms to avoid mid-rename reads (UI work, out of scope here but noted).
- No fs locks in v1. The rules file documents: "do not run two agents against the same session".

## Schema Versioning

- `schemaVersion: 1` in both files.
- Reader: version mismatch → hard error `unsupported_schema_version: <n>`. No migrations in v1.

## Error Taxonomy

All extend `FlowbuilderError`. Every error includes `sessionId` and the absolute path involved.

| Error                              | Raised by              | Outcome                                  |
| ---------------------------------- | ---------------------- | ---------------------------------------- |
| `FlowbuilderSessionMissingError`   | `preRun`               | Harness aborts before agent starts.      |
| `FlowbuilderSchemaError`           | preRun, set_state read | preRun: abort. set_state: tool error.    |
| `FlowbuilderRefIntegrityError`     | set_state              | Tool error returned to agent.            |
| `FlowbuilderIOError`               | any                    | Wraps fs failures with cause.            |
| `FlowbuilderUnsupportedVersion`    | preRun                 | Hard error.                              |

Tool errors are returned as `{ ok: false, error: "<code>: <detail>" }` so the agent can act on them. Plugin errors thrown from `preRun` propagate up and abort the run.

## Testing

Test runner: existing `vitest` setup mirrored from `packages/rote`. All tests use real tmpdirs (`os.tmpdir()`), no fs mocks.

### Unit — `packages/flowbuilder/test/`

- `schema.test.ts`: accept fixtures for each node type; reject malformed inputs with the expected Zod path.
- `session.test.ts`: `SessionManager` round-trip, atomic write under tmpdir, missing-session throw, manifest `updatedAt` bump.
- `tools.test.ts`: each tool's success path + each error path. Assert disk state after `set_state`.

### Integration — `packages/flowbuilder/test/plugin.integration.test.ts`

- Spin up the plugin against a tmpdir.
- Simulate `preRun` → tool calls → final state.
- Assert manifest and state on disk after the simulated run.

### Existing tests

- `docs/smoke-tests/...`: update to create a temp session + base dir before invoking `flow-build run`. Pass both required flags. Add helper `setupTestSession(tmpDir)` returning `{ sessionId, baseDir }`.
- Rote integration tests in the harness: same helper.

There is no opt-out env var; tests must always set up a session.

## Dependencies

- `nanoid` — id generation. Owned by the Electron main process at session-create time. `packages/flowbuilder` does not depend on it at runtime; tests use it as a dev dep for fixture session ids.
- `zod` (already a workspace dep) — schema validation.
- `zod-to-json-schema` (new) — converts Zod schemas to JSON Schema for the Cursor SDK tool definitions.

## Out Of Scope

- UI rendering of `state.json`.
- IPC channel between Electron main and renderer.
- File-watch strategy in the renderer (chokidar config, debounce specifics).
- Multi-process locking. Contract is one agent per session.
- Power-loss durability of atomic writes.
- State migrations across `schemaVersion` bumps. v1 only; mismatch is a hard error.
- Runtime execution semantics — how the agent actually walks the graph end-to-end at the user's request.
- Output passing / `$ref` resolution between nodes. The agent maps upstream output to downstream params at runtime; no formal wiring schema in the file.
- Process model decision — one long-lived agent vs one process per turn. Electron's call, not the harness's.
- Session lifecycle on the Electron side — `createSession` IPC, subprocess spawn, kill, cleanup.
- Listing or picking sessions in the UI. No `list_sessions` tool; the agent always operates in a single, pre-bound session.

These each get their own spec when needed.
